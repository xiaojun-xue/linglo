# 解决方案设计（一）：精简版 IPD 流程设计

> 项目：IPD-ProjectPlatform | 版本：v1.0 | 日期：2026-04-10
> 设计原则：**保留华为 IPD 精髓，去除过度工程化，让中小企业用得上、用得起**

---

## 一、设计理念

### 1.1 华为 IPD 原版 vs 本项目精简版

| 维度 | 华为 IPD 原版 | 本项目精简版 |
|------|-------------|-------------|
| 阶段数量 | 6-8 个阶段 | 4 个核心阶段 |
| Gate 数量 | 6-8 个门禁 | 4 个决策门（精简但有效）|
| 评审文件 | 20+ 份文档要求 | 5-8 份核心文档 |
| 角色数量 | 10+ 种角色 | 6 种核心角色 |
| 流程复杂度 | 高（适合大企业）| 低（适配中小企业）|
| 实施周期 | 6-24 个月 | 1-4 周上手 |
| 适用规模 | 500人+ 研发团队 | 50-500人团队 |

### 1.2 精简版 IPD 核心原则

**做减法，不做减法功：**
- ❌ 不要：复杂的产品组合管理（PPM）
- ✅ 要：简化的多项目管理
- ❌ 不要：完整的财务分析模型
- ✅ 要：项目预算和 ROI 估算
- ❌ 不要：市场管理（MM）全套
- ✅ 要：需求池管理 + 优先级排序

## 三、四个核心阶段详解

### 阶段 1：概念阶段（Concept）

**阶段目标：** 确定要做什么产品/项目，评估是否值得投入

**核心活动：**
```
1. 市场需求收集
   ├── 客户需求（客服反馈、访谈）
   ├── 销售需求（市场反馈）
   ├── 竞品分析（直接竞品、对标产品）
   └── 内部需求（技术储备、战略布局）

2. 需求分析与筛选
   ├── 需求池建立（所有输入汇集）
   ├── 需求分类（功能/技术/体验）
   ├── 需求初步评估（市场价值 + 技术可行性）
   └── 需求优先级初排（Kano模型 / MoSCoW）

3. 商业初步分析
   ├── 目标市场规模
   ├── 预期收益估算
   ├── 初步成本估算
   └── ROI 初步评估

4. 产品概念定义
   ├── 产品定位（卖给谁、解决什么问题）
   ├── 产品价值主张（差异化卖点）
   ├── 目标市场描述
   └── 初步产品范围（Scope）
```

**阶段产出文档：**
| 文档 | 必须/可选 | 说明 |
|------|---------|------|
| 市场需求说明书 | 必须 | 需求来源 + 初步分析 |
| 产品概念文档 | 必须 | 产品定位 + 价值主张 + 初步范围 |
| 商业价值评估表 | 可选（建议）| 简单 ROI 估算 |
| 概念阶段报告 | 必须 | 汇总文档，提交 CDCP 评审 |

**CDCP 评审（Concept to Develop Check Point）**
- 评审问题：我们是否要启动这个项目？
- 评审内容：市场需求是否真实、产品概念是否清晰、是否值得投入
- 评审结果：通过 / 拒绝 / 搁置 / 需修改后重审
- 评审权限：产品负责人 + 技术负责人 + 项目发起人

---

### 阶段 2：计划阶段（Plan）

**阶段目标：** 制定详细计划，确保项目可执行

**核心活动：**
```
1. 产品规划
   ├── 详细需求分析（User Story Mapping）
   ├── 功能范围确认（In Scope / Out of Scope）
   ├── 需求优先级最终确认（投票/MoSCoW）
   └── 产品路标规划（里程碑）

2. 技术规划
   ├── 技术方案初步设计
   ├── 技术选型评估
   ├── 技术风险识别
   ├── 依赖关系分析（内部依赖 + 外部依赖）
   └── 架构设计评审（Architecture Review）

3. 项目规划
   ├── WBS 任务分解（到 2-4 周 Sprint）
   ├── 工时估算（敏捷估算扑克牌）
   ├── 资源分配（人员 + 设备 + 预算）
   ├── 进度计划（甘特图）
   └── 风险应对计划

4. 质量规划
   ├── 质量目标定义
   ├── 测试策略制定
   ├── 验收标准定义
   └── 质量评审点规划

5. 商业计划细化
   ├── 详细成本预算
   ├── 上市计划
   ├── 定价策略
   └── 盈亏平衡分析
```

**阶段产出文档：**
| 文档 | 必须/可选 | 说明 |
|------|---------|------|
| 产品需求规格说明书（PRD）| 必须 | 完整需求定义 |
| 技术设计方案 | 必须 | 架构 + 技术选型 |
| 项目计划书 | 必须 | 进度 + 资源 + 风险 |
| 测试计划 | 必须 | 质量保障方案 |
| 商业计划书 | 可选 | 详细财务分析 |
| 计划阶段报告 | 必须 | 汇总文档，提交 PDCP 评审 |

**PDCP 评审（Plan to Develop Check Point）**
- 评审问题：我们是否准备好进入开发阶段？
- 评审内容：计划是否完整可行、风险是否可控、资源是否到位
- 评审结果：通过 / 拒绝 / 条件通过（需补充） / 需修改后重审
- 评审权限：项目经理 + 技术负责人 + 产品负责人 + QA 负责人

---

### 阶段 3：开发阶段（Develop）

**阶段目标：** 交付满足质量要求的产品

**核心活动：**
```
1. Sprint 1（如果项目较大，可多 Sprint）
   ├── Sprint Planning（周/双周）
   ├── 每日站会（Daily Standup）
   ├── 持续开发
   ├── 代码评审（Code Review）
   ├── 持续集成（CI）
   └── Sprint Review & Retrospective

2. 质量保障
   ├── 单元测试（Unit Test）
   ├── 集成测试（Integration Test）
   ├── 系统测试（System Test）
   ├── 验收测试（UAT）
   ├── 性能测试（如需要）
   └── 安全测试（如需要）

3. 风险管理
   ├── 周度风险评审
   ├── 风险应对执行
   └── 新风险识别与登记

4. 需求变更管理
   ├── 变更申请（Change Request）
   ├── 变更评审（Impact Assessment）
   ├── 变更决策
   └── 计划更新

5. TR4 技术评审（Technical Review 4）
   ├── 技术方案实现验证
   ├── 设计符合性检查
   ├── 测试覆盖率评审
   └── 遗留问题评审
```

**阶段产出文档：**
| 文档 | 必须/可选 | 说明 |
|------|---------|------|
| Sprint 报告 | 必须 | 每个 Sprint 输出 |
| 测试报告 | 必须 | 各轮测试结果 |
| 缺陷报告 | 必须 | Bug 追踪 |
| 阶段进展报告 | 必须 | 里程碑进展 |
| 变更记录 | 可选 | 如有变更 |

**TR4 技术评审**
- 评审问题：技术实现是否满足设计要求？
- 评审内容：代码质量、测试结果、技术债务、遗留问题
- 评审结果：通过（可进入发布准备）/ 需整改 / 需返工
- 评审权限：技术负责人 + 架构师 + QA

---

### 阶段 4：发布阶段（Launch）

**阶段目标：** 成功交付产品并持续运维

**核心活动：**
```
1. 发布准备
   ├── 发布检查清单（Release Checklist）
   ├── 最终测试（Release Testing）
   ├── 文档完善（用户手册 + 运维文档）
   ├── 培训材料准备
   └── 部署计划制定

2. 正式发布
   ├── 生产环境部署
   ├── 灰度发布（如适用）
   ├── 全量发布
   └── 发布验证

3. 运营支持
   ├── 运营监控（系统 + 业务）
   ├── 问题响应
   ├── 缺陷修复
   └── 持续改进

4. 项目收尾
   ├── 项目总结（复盘）
   ├── 经验教训沉淀
   ├── 文档归档
   ├── 资源释放
   └── 项目验收
```

**阶段产出文档：**
| 文档 | 必须/可选 | 说明 |
|------|---------|------|
| 发布报告 | 必须 | 发布内容 + 结果 |
| 用户手册 | 必须 | 最终用户使用文档 |
| 运维手册 | 必须 | 运维团队使用文档 |
| 项目总结报告 | 必须 | 复盘文档，经验沉淀 |
| ADCP 评审材料 | 必须 | 提交 ADCP 评审 |

**ADCP 评审（Available to Deploy Check Point）**
- 评审问题：产品是否可以正式发布？
- 评审内容：质量是否达标、业务目标是否达成、运营准备是否充分
- 评审结果：可以发布 / 需整改 / 延迟发布
- 评审权限：产品负责人 + 技术负责人 + 项目经理 + 业务代表

## 四、全流程支撑模块

### 4.1 需求管理模块

```
需求生命周期：
┌─────────┐    ┌──────────┐    ┌──────────┐    ┌─────────┐    ┌────────┐
│ 需求收集 │───▶│ 需求分析 │───▶│ 需求评审 │───▶│ 开发实现 │───▶│ 验证确认 │
└─────────┘    └──────────┘    └──────────┘    └─────────┘    └────────┘
     ↑              ↓               ↓              ↓
     └── 需求池 ◄───┴───────────────┘              │
                                                   ▼
                                            需求变更流程
```

**需求属性：**
- 编号（自动生成）
- 标题
- 需求类型（功能/技术/体验/缺陷）
- 需求来源（客户/销售/内部/竞品）
- 优先级（必须/重要/应该/可以/不要）
- 估算（故事点/人天）
- 状态（收集/分析/设计/开发/测试/发布）
- 负责人
- 关联需求
- 关联任务

### 4.2 立项管理模块

**立项申请流程：**
```
1. 发起立项申请
   ├── 选择立项级别（一般/重要/重大）
   ├── 填写立项信息
   │   ├── 项目名称
   │   ├── 项目背景与目标
   │   ├── 预期收益
   │   ├── 资源需求
   │   ├── 初步时间计划
   │   └── 风险评估
   ├── 上传立项文档
   └── 选择审批人

2. 审批流程（可配置）
   ├── 一般项目：产品负责人审批
   ├── 重要项目：产品负责人 + 技术负责人
   └── 重大项目：管理层 + 产品 + 技术 + 财务

3. 立项决议
   ├── 批准：正式进入概念阶段
   ├── 拒绝：退回，补充信息
   ├── 搁置：暂不启动
   └── 合并：与其他项目合并

4. 立项变更
   ├── 范围变更（需重新评审）
   ├── 预算变更（需追加审批）
   └── 终止项目（需正式审批）
```

### 4.3 评审管理模块

**评审类型：**
| 评审类型 | 时机 | 评审内容 | 评审人 | 决策选项 |
|---------|------|---------|--------|---------|
| CDCP | 概念→计划 | 产品概念、市场价值 | 产品+技术+发起人 | 通过/拒绝/搁置 |
| PDCP | 计划→开发 | 项目计划、风险、资源 | 项目+技术+QA | 通过/条件通过/拒绝 |
| TR4 | 开发中 | 技术实现质量 | 技术+QA | 通过/整改/返工 |
| ADCP | 开发→发布 | 产品发布就绪性 | 产品+技术+业务 | 发布/整改/延迟 |

**评审流程：**
```
1. 评审发起
   ├── 评审材料准备
   ├── 评审通知发送
   └── 评审时间确认

2. 评审会议（可简化）
   ├── 评审主持人介绍
   ├── 汇报人陈述
   ├── 评委提问与讨论
   └── 评委决策投票

3. 评审输出
   ├── 评审结论（通过/拒绝/条件通过）
   ├── 遗留问题清单
   ├── 整改要求
   └── 评审记录归档
```

### 4.4 风险与问题管理

**风险管理流程：**
```
风险识别 ──▶ 风险评估（概率 × 影响）──▶ 风险应对计划 ──▶ 风险监控 ──▶ 风险解决/升级
```

**风险等级矩阵：**
| 等级 | 概率 × 影响 | 处理策略 |
|------|------------|---------|
| 高风险 🔴 | ≥15 | 必须制定应对计划，由管理层关注 |
| 中风险 🟡 | 8-14 | 需要制定应对计划，项目经理关注 |
| 低风险 🟢 | <8 | 日常监控，可接受风险 |

### 4.5 质量管理模块

**质量保障活动：**
```
需求设计 ──▶ 代码实现 ──▶ 单元测试 ──▶ 集成测试 ──▶ 系统测试 ──▶ UAT ──▶ 发布
     │                                    │
     └────── 质量评审点（每个阶段门）◄─────┘
```

**缺陷管理流程：**
```
缺陷发现 ──▶ 缺陷登记 ──▶ 缺陷分配 ──▶ 缺陷修复 ──▶ 缺陷验证 ──▶ 缺陷关闭
     │
     └── 缺陷严重等级（P0-P3）
```

## 五、流程可配置性设计

**原则：中小企业团队规模各异，流程需要灵活适配**

### 5.1 流程模板

| 模板名称 | 适用场景 | 阶段数量 | 评审严格度 |
|---------|---------|---------|-----------|
| 精简版 | 10-50人团队 | 3阶段 | 轻量评审 |
| 标准版 | 50-200人团队 | 4阶段 | 标准评审 |
| 完整版 | 200-500人团队 | 4阶段 | 严格评审 |

### 5.2 评审简化选项

| 选项 | 说明 |
|------|------|
| 会议评审 | 线下/线上会议，正式评审 |
| 文档评审 | 在线文档评论+审批流 |
| 简化评审 | 仅负责人确认即可 |
| 免评审 | 低风险项目可跳过 |

### 5.3 评审流配置

**管理员可配置：**
- 各阶段评审人（角色/人员）
- 评审通过条件（必须全部通过/多数通过）
- 评审通知方式（站内/邮件/钉钉/飞书）
- 评审超时提醒时间

## 八、后续设计关联
