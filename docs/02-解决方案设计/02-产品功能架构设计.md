# 解决方案设计（二）：产品功能架构设计

> 项目：IPD-ProjectPlatform | 版本：v1.0 | 日期：2026-04-10
> 基于：精简版 IPD 流程设计 | 目标：中小企业 IPD 研发管理

---

## 一、产品愿景与定位

### 1.1 产品愿景

> **让每一家中小企业都能享受 IPD 体系带来的研发管理升级**

### 1.2 产品定位

**一句话定位：**
> 一款专为中小企业设计的轻量级 IPD 研发管理平台

**三要素定位：**
| 要素 | 描述 |
|------|------|
| 为谁 | 50-500人研发团队的 CTO、PMO、项目经理、开发负责人 |
| 解决什么问题 | IPD 流程落地难、学习成本高、工具水土不服 |
| 提供什么价值 | 完整 IPD 核心流程 + 现代化体验 + 轻量化实施 |

### 1.3 核心价值主张

| # | 价值主张 | 说明 |
|---|---------|------|
| 1 | **真正的 IPD，不是伪敏捷** | 完整 TR/DCP/Gate 评审流程，不是简单任务管理 |
| 2 | **开箱即用的 IPD 模板** | 预置最佳实践，无需从零配置 |
| 3 | **一天上手，一周用熟** | 现代化 UI，让团队愿意用 |
| 4 | **数据驱动的决策** | 内置研发效能分析仪表盘 |
| 5 | **私有化部署，数据自主** | 支持私有化，符合信创要求 |

```
│  ┌────────▼─────────┐                                          │
│  │   需求详情页      │  ← 查看/编辑单条需求                      │
│  │  (Requirement)  │    关联任务、评审、文档                   │
│  └────────┬─────────┘                                          │
│           │                                                      │
│  ┌────────▼─────────┐    ┌──────────────┐                      │
│  │   需求分析工具    │◄───│  需求看板     │                      │
│  │  (Req Analysis) │    │  (Kanban)    │                      │
│  │  KANO/MoSCoW    │    │  收集→分析→   │                      │
```

**需求属性设计：**
| 属性 | 类型 | 说明 | 必填 |
|------|------|------|------|
| 需求编号 | 自动 | REQ-年份-序号 | 自动 |
| 需求标题 | 文本 | 简明扼要描述 | ✅ |
| 需求类型 | 枚举 | 功能/技术/体验/缺陷 | ✅ |
| 需求来源 | 枚举 | 客户/销售/内部/竞品/战略 | ✅ |
| 需求描述 | 富文本 | 详细描述（支持附件）| ✅ |

| 优先级 | 枚举 | P0(必须)/P1(重要)/P2(一般)/P3(可选) | ✅ |
| 估算 | 数字 | 故事点/人天 | |
| 所属产品 | 关联 | 关联产品线 | ✅ |
| 关联需求 | 关联 | 依赖/被依赖关系 | |
| 关联项目 | 关联 | 分配到哪个项目 | |
| 状态 | 枚举 | 收集/分析/评审/开发/测试/发布/关闭 | 自动 |
| 提出人 | 人员 | 需求提出者 | |
| 负责人 | 人员 | 需求负责人 | |
| 创建时间 | 时间 | 自动记录 | 自动 |
| 更新时间 | 时间 | 自动更新 | 自动 |

**评审类型与配置：**
| 评审类型 | 触发时机 | 评审材料 | 评委配置 | 决策选项 |
|---------|---------|---------|---------|---------|
| CDCP | 概念→计划 | 产品概念文档、商业价值表 | 产品+技术+发起人 | 通过/拒绝/搁置 |
| PDCP | 计划→开发 | PRD、技术方案、项目计划 | 项目+技术+QA | 通过/条件通过/拒绝 |
| TR4 | 开发中 | 代码、测试报告 | 技术+QA | 通过/整改/返工 |
| ADCP | 开发→发布 | 发布检查表、测试报告 | 产品+技术+业务 | 发布/整改/延迟 |

**评审流程状态机：**
```

       │      │         │      │

       │      │         │
       │      │         └──▶ 拒绝 ──▶ 可申诉

       └──▶ 已撤回
```
